# find_cenhaps

`find_cenhaps.py` identifies candidate centromeric haplotype marker k-mers.
It can start from either a physical k-mer map generated by `map_kmers_ac.2.1.py`
or from a reference FASTA plus core-centromere coordinates.

The program is not limited to Arabidopsis or to five chromosomes. It discovers
the centromeres to compare from the coordinate file, so a genome with any
number of chromosome-level core centromeres can be analyzed as long as the map
and coordinate chromosome names agree.

The core idea is:

1. Count each k-mer's hits in every core centromere listed in the coordinate
   file.
2. Assign the k-mer to the core centromere with the highest hit count.
3. Keep only k-mers that are strongly biased to that one core centromere.
4. Collapse overlapping target hits into blocks so adjacent overlapping k-mers
   around the same polymorphism do not inflate the cenhap strength.
5. Group selected k-mers into non-redundant cenhap units when they overlap in
   any assigned core-centromere location.

## Input Modes

### Physical Map

When available, provide a k-mer map with `--map-tsv`. This is the most direct
mode and preserves the original physical mapping evidence.

The k-mer map must be a tab-separated file with at least these columns:

```text
k-mer    chr    pos    strand
```

### FASTA + Coordinates

If a physical map is not available, omit `--map-tsv` and provide `--fasta`.
The script will generate a core-CEN marker map from the FASTA and coordinate
file, then run the same cenhap analysis on that generated map.

Default FASTA-generation thresholds:

- `--kmer-size 23`
- `--kmer-step 10`
- `--min-cen-count 10`
- `--max-outside-ratio 0.25`

The generated-map specificity filter keeps a k-mer when:

```text
core_CEN_count >= min_cen_count
outside_CEN_count <= core_CEN_count * max_outside_ratio
```

Reverse complements are collapsed by default. Use `--no-canonical` to keep
strand-specific k-mers.

The generated map and its stats are written beside the normal outputs:

- `*.generated_core_kmer_map.tsv`
- `*.generated_core_kmer_map.stats.tsv`

## Coordinate Contract

The coordinate file may use either a compact core-centromere format:

```text
Chr    Left       Right
1      15000000   17000000
2      4800000    6300000
```

or a region-table format:

```text
chr    region       start      end
Chr1   centromere   15000000   17000000
Chr2   centromere   4800000    6300000
```

The optional FASTA is used for chromosome-length validation and reporting. The
program assumes one named core-centromere interval per chromosome/scaffold
label. If a genome has multiple separate core-centromere intervals on the same
sequence, those intervals should be given distinct labels or the script should
be extended to avoid label collisions.

## Arabidopsis Col-PEK Examples

With an existing physical k-mer map:

```bash
./find_cenhaps.py \
  --map-tsv /Users/lucacomai/Documents/Data_analysis_23/new_tec/10_lib_mpiz_hamb_busco/map_dft_std_all_kmers.tsv \
  --coords /Users/lucacomai/Documents/Data_analysis_23/Genomes/Col-PEK1.5/colpek_cen_coor.tsv \
  --fasta /Users/lucacomai/Documents/Data_analysis_23/Genomes/Col-PEK1.5/file1.Col-PEK1.5_Chr1-5_20220523.fasta \
  --prefix outputs/colpek_cenhaps/colpek_cenhaps
```

Without a physical k-mer map:

```bash
./find_cenhaps.py \
  --coords /Users/lucacomai/Documents/Data_analysis_23/Genomes/Col-PEK1.5/colpek_cen_coor.tsv \
  --fasta /Users/lucacomai/Documents/Data_analysis_23/Genomes/Col-PEK1.5/file1.Col-PEK1.5_Chr1-5_20220523.fasta \
  --prefix outputs/colpek_cenhaps_from_fasta/colpek_cenhaps_from_fasta
```

Default selection thresholds:

- `--min-target-hits 3`
- `--max-other-core-hits 1`
- `--min-target-core-fraction 0.90`
- `--min-target-enrichment 3.0`
- `--kmer-step 10` when generating from FASTA
- `--min-cen-count 10` when generating from FASTA
- `--max-outside-ratio 0.25` when generating from FASTA
- `--merge-gap 0`
- `--bin-size 200000`
- `--window-size 100000`
- `--window-step 25000`
- `--min-shared-cens 2`
- `--min-shared-hits-per-cen 1`

Main outputs:

- `*.kmer_summary.tsv`: selected and near-threshold k-mers with one count
  column per core centromere.
- `*.selected_kmers.txt`: one selected k-mer per line.
- `*.assigned_core_map.tsv`: selected map rows inside the assigned core CEN.
- `*.cenhap_blocks.tsv`: overlap-collapsed physical blocks.
- `*.cenhap_units.tsv`: non-redundant k-mer groups connected by overlap.
- `*.cenhap_unit_size_summary.tsv`: per-centromere CU size summaries, where
  size is primarily distinct selected k-mers per CU.
- `*.cenhap_unit_size_distribution.tsv`: bucketed CU size distribution.
- `*.cenhap_unit_size_distribution.svg`: visual CU size distribution.
- `*.cenhap_strength.tsv`: per-centromere raw selected k-mer counts and
  collapsed unit/block counts.
- `*.cenhap_strength_histogram.svg`: visual comparison of the main strength
  metrics across all core centromeres in the coordinate file.
- `*.cen_shared_kmers.tsv`: CRISPR-facing diagnostic table of k-mers present
  in at least `--min-shared-cens` core CENs. It includes all-CEN common k-mers
  and k-mers shared by exact CEN subsets.
- `*.cen_shared_kmer_sets.tsv`: summary counts for each exact shared CEN set.
- `*.cen_relatedness.idf_weighted_pairs.tsv`: pairwise CEN relatedness using
  IDF-weighted Jaccard on core-CEN k-mer counts.
- `*.cen_relatedness.idf_weighted_matrix.tsv`: matrix form of the same
  relatedness scores.
- `*.cenhap_bins.tsv`: fixed-bin counts of cenhap-defining k-mers along each
  core CEN. The default bin size is 200 kb.
- `*.cenhap_bin_counts.svg`: simple fixed-bin plot of distinct selected
  cenhap-defining k-mers per bin.
- `*.cenhap_windows.tsv`: running-window local strength along each core CEN.
- `*.cenhap_window_dispersion.tsv`: per-centromere summaries of how dispersed
  or locally concentrated the windowed signal is.
- `*.cenhap_local_strength.svg`: optional older overlapping-window visual
  track, written only with `--write-window-plot`.
- `*.stats.txt`: run parameters and summary statistics.

The most useful first-pass cenhap strength column is `cenhap_strength_units` in
`*.cenhap_strength.tsv`: it counts distinct selected k-mer groups after
collapsing overlapping k-mers. The `selected_distinct_kmers` column is
intentionally reported too, because it is useful for judging how much raw
sequence evidence was collapsed.

For CU size, start with `*.cenhap_unit_size_summary.tsv` and
`*.cenhap_unit_size_distribution.svg`. The primary CU size is
`distinct_kmers`, the number of selected centromere-biased k-mers grouped into
that CU. The unit table also reports `target_map_hits` and `physical_blocks`,
which are useful secondary size/support measures.

For shared-kmer diagnostics, start with `*.cen_shared_kmers.tsv`. A k-mer is
counted as present in a CEN when it has at least
`--min-shared-hits-per-cen` hits in that CEN. The `present_cens` column gives
the exact CEN set, and `shared_class` distinguishes `all_core_cens` from
`subset_core_cens`. The `idf_weight` column keeps common k-mers in the
relatedness calculation but gives less influence to k-mers found in many CENs.
This is useful for CRISPR design because the table explicitly separates
candidate all-CEN targets from subset-specific targets.

For regional interpretation, start with `*.cenhap_bin_counts.svg` and
`*.cenhap_bins.tsv`. These use non-overlapping fixed bins and simply count how
many selected cenhap-defining k-mers occur in each interval. The running-window
dispersion table is still useful for quantifying broad versus localized signal:
broad signal across many windows suggests a dispersed cenhap; a high maximum
window with a high Gini/top-10% fraction points to localized divergence islands.
